# 커밋 규약

이 프로젝트의 모든 커밋은 아래 규약을 따른다. 커밋은 단순한 저장 행위가 아니라 **작업의 논리 단위를 기록하는 1차 매체** 이며, 이후 PR 작성, 리뷰 응답, 세션 간 컨텍스트 인계가 모두 이 기록을 재료로 삼는다. 커밋 메시지가 부실하면 그 공백을 `git diff` 재독으로 메워야 하고, 이는 AI 세션의 컨텍스트 낭비로 직결된다.

## 1. 핵심 원칙

- **Phase 는 리뷰 단위이지 커밋 단위가 아니다.** 하나의 Phase 는 보통 여러 개의 커밋으로 구성된다.
- **`git log --oneline` 만 읽어도 "무엇을 / 어디에 / 어떻게" 가 드러나야 한다.** diff 를 다시 읽어야 의도가 복원된다면 커밋 메시지가 부족한 것이다.
- **쪼개기 자체가 목표가 아니다.** 목표는 "기록의 해상도" 다. 무의미하게 잘게 쪼갠 커밋(`fix typo` 연발) 은 오히려 노이즈가 된다.

## 2. 커밋 단위 기준

적정 단위는 **"독립적으로 revert 가능한 최소 논리 단위"** 이다. 판단 질문:

> "이 커밋 하나만 되돌렸을 때, 코드가 일관된 중간 상태를 유지하는가?"

- ❌ 너무 큼: "Phase 5 전체" — 서로 독립적인 파일/패턴 치환이 한 커밋에 섞임
- ❌ 너무 잘음: "Sidebar width 변경" + "해당 import 정리" + "prettier 적용" — 하나의 작업이 3개로 흩어짐
- ✅ 적정: "Sidebar layout 치수 토큰화" — 관련 import/포맷 포함, 다른 컴포넌트는 별도 커밋

## 3. 커밋 메시지 포맷

```
<type>(<scope>): <대상> — <변경의 성격>

<선택: 비자명한 결정이나 배경 1~2줄>
```

### 3-1. type (필수)

| type | 용도 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변화 없는 코드 구조 개선 |
| `style` | 코드 포맷팅, 공백, 세미콜론 등 (로직 변경 없음) |
| `docs` | 문서 / 주석 수정 |
| `test` | 테스트 코드 추가 / 수정 |
| `chore` | 빌드, 패키지, 설정 등 기타 |

### 3-2. scope (필수)

기능 모듈명 또는 최상위 폴더명.

- 기능 / 프로그램 단위: `theme`, `folder-program`, `desktop`, `sidebar`
- 영역: `api`, `store`, `hooks`
- 문서 / 설정: `commit-convention`, `worker-create-pr`, `agents` 등

scope 가 모호하거나 여러 영역에 걸친다면 **커밋 단위가 너무 큰 것** 이다. 쪼개는 것을 먼저 고려한다.

### 3-3. 대상 + 성격

- **대상**: 어떤 파일 / 컴포넌트 / 영역을 건드렸는지 — "어디에"
- **변경의 성격**: "어떻게" 바꿨는지 한 줄 요약

### 3-4. 예시

- ✅ `refactor(theme): Sidebar 고정 width 를 sizes.sidebarWidth 토큰으로 치환`
- ✅ `fix(folder-program): 폴더명 인라인 편집 Esc 취소 시 상태 복원`
- ✅ `test(theme-tokens): spacing 토큰 렌더링 회귀 테스트 추가`
- ❌ `refactor: theme 정리` — scope 너무 넓고 대상 / 성격 부재
- ❌ `Phase 5 진행` — type / scope / 내용 전부 부재
- ❌ `fix: 버그 수정` — 대상 / 성격 부재

### 3-5. 본문 (선택)

본문은 **비자명한 결정** 에만 작성한다. "왜 이 토큰 이름을 골랐는가", "왜 이 파일만 제외했는가" 같은 선택 근거가 있을 때만. 자명한 변경이면 제목만으로 충분하다.

## 4. 수정 커밋: 스택 방식

사용자 확인 이후 수정 요청이 들어오면 **새 커밋을 쌓아서** 대응한다. `git commit --amend` / `git rebase -i squash` / `git reset` 으로 히스토리를 재작성하지 않는다.

### 4-1. 이유

수정 커밋은 **"판단이 교정된 사실"** 을 기록한다. 미래 세션의 AI 가 log 를 시간순으로 읽으면 "원 의도 → 교정 사유 → 최종 결과" 가 자연스럽게 복원된다. squash / amend 는 이 맥락을 파괴한다.

### 4-2. 메시지 규약

수정 커밋임이 제목 또는 본문에서 명시되어야 한다.

- 제목 예: `fix(theme): Sidebar 토큰명 교정 (sizes.sidebar → sizes.sidebarWidth)`
- 본문 예: `앞선 a1b2c3d 커밋에서 토큰명 오타. 정식 이름으로 교정.`

### 4-3. 예외: 방금 쌓은 커밋의 메시지만 교정

코드 변경 없이 **방금 쌓은 커밋의 메시지만** 교정할 때는 `git commit --amend -m` 을 허용한다. 아직 다른 커밋이 위에 쌓이지 않은 상태에서만 적용된다. 이 예외의 목적은 오타나 형식 누락을 즉시 바로잡는 것이며, 코드 변경을 숨기는 용도가 아니다.
