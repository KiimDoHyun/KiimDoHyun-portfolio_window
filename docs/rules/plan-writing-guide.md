# 설계 문서 작성 가이드

설계 문서를 작성할 때 다음 패턴을 따른다.

## 1. Phase별 독립 실행 가능하게 설계

각 Phase는 **자기 완결적**으로 작성한다. 이전 대화 맥락 없이 문서만 읽고 바로 실행 가능해야 한다.

Phase마다 다음을 명시한다:

- **입력**: 이 Phase를 시작하기 위해 필요한 선행 조건
- **작업 내용**: 구체적인 변경 항목 (체크리스트) — 각 항목은 커밋 1개에 대응하도록 설계한다. 자세한 기준은 [6. 커밋 단위와 기록 원칙](#6-커밋-단위와-기록-원칙) 참조
- **완료 조건**: 이 Phase가 끝났다고 판단하는 기준

## 2. 배경(Why)과 규칙을 문서에 내장

매 세션마다 반복 지시하지 않아도 되도록, 문서 자체에 포함시킨다:

- **배경**: 왜 이 작업이 필요한지
- **설계 규칙**: 작업 중 지켜야 할 원칙 (예: "Shell에는 로직을 넣지 않는다")
- **성공 기준 (Definition of Done)**: 전체 작업의 완료 조건

## 3. 진행 상태 추적

`- [ ]` / `- [x]` 체크리스트를 사용하여 진행 상태를 문서에 직접 반영한다. 새 세션에서 "이어서 해줘"가 가능해야 한다.

## 4. Phase 완료 시 검증과 회고

각 Phase가 완료되면 다음 두 가지를 문서에 남긴다.

### 4-1. 수동 검증 항목

사용자가 직접 확인해야 할 테스트 시나리오를 구체적으로 작성한다. 나중에 자동화 테스트의 기반 데이터로 활용된다.

```markdown
#### 수동 검증
- [ ] 바탕화면 아이콘 더블클릭 → 해당 프로그램 창이 열린다
- [ ] 폴더 열기 → 하위 파일 목록이 정상 표시된다
- [ ] 상태바 → 프로젝트 목록이 올바르게 노출된다
```

### 4-2. Phase 회고

주관적 평가가 아니라 **검증 가능한 사실 기반 관찰**만 남긴다. 사용자가 검토 후 확정한다.

```markdown
<!-- 나쁜 예 (검증 불가) -->
> 패턴이 잘 적용되었다

<!-- 좋은 예 (검증 가능) -->
> fsState를 통째로 넘기다가 selector로 변경 → Shell 코드가 12줄 → 5줄로 줄었다
```

## 5. 프로젝트 회고

모든 Phase 완료 후, Phase별 회고를 취합하여 전체 회고를 작성한다:

- **잘된 점**: 다음에도 유지할 패턴
- **개선할 점**: 다음에 보완할 사항
- **향후 과제**: 이 작업에서 파생된 후속 작업

반복 적용 가능한 교훈은 본 가이드(plan-writing-guide.md)에 반영하여 다음 프로젝트 설계에 자동으로 적용되도록 한다.

## 6. 커밋 단위와 기록 원칙

각 체크리스트 항목은 **커밋 1개**에 대응한다. 커밋은 단순한 저장 행위가 아니라 **작업의 논리 단위를 기록하는 1차 매체**이며, 이후 PR 작성, 리뷰 응답, 세션 간 컨텍스트 인계가 모두 이 기록을 재료로 삼는다. 커밋 메시지가 부실하면 그 공백을 `git diff` 재독으로 메워야 하고, 이는 AI 세션의 컨텍스트 낭비로 직결된다.

### 6-1. 핵심 원칙

- **Phase는 리뷰 단위이지 커밋 단위가 아니다.** 하나의 Phase는 보통 여러 개의 커밋으로 구성된다.
- **`git log --oneline` 만 읽어도 "무엇을 / 어디에 / 어떻게" 가 드러나야 한다.** diff를 다시 읽어야 의도가 복원된다면 커밋 메시지가 부족한 것이다.
- **쪼개기 자체가 목표가 아니다.** 목표는 "기록의 해상도"다. 무의미하게 잘게 쪼갠 커밋(`fix typo` 연발)은 오히려 노이즈가 된다.

### 6-2. 커밋 단위 기준

적정 단위는 **"독립적으로 revert 가능한 최소 논리 단위"** 이다. 판단 질문:

> "이 커밋 하나만 되돌렸을 때, 코드가 일관된 중간 상태를 유지하는가?"

- ❌ 너무 큼: "Phase 5 전체" — 서로 독립적인 파일/패턴 치환이 한 커밋에 섞임
- ❌ 너무 잘음: "Sidebar width 변경" + "해당 import 정리" + "prettier 적용" — 하나의 작업이 3개로 흩어짐
- ✅ 적정: "Sidebar layout 치수 토큰화" — 관련 import/포맷 포함, 다른 컴포넌트는 별도 커밋

### 6-3. 커밋 메시지 포맷

```
<type>(<scope>): <대상> — <변경의 성격>

<선택: 비자명한 결정이나 배경 1~2줄>
```

- **type**: `feat` / `fix` / `refactor` / `chore` / `docs` / `test` 중 하나
- **scope**: 기능 모듈명 또는 최상위 폴더 (예: `theme`, `folder-program`, `desktop`)
- **대상**: 어떤 파일/컴포넌트/영역을 건드렸는지 — "어디에"
- **변경의 성격**: "어떻게" 바꿨는지 한 줄 요약

**예시:**

- ✅ `refactor(theme): Sidebar 고정 width를 sizes.sidebarWidth 토큰으로 치환`
- ✅ `fix(folder-program): 폴더명 인라인 편집 Esc 취소 시 상태 복원`
- ❌ `refactor: theme 정리` — 대상/성격 모두 부재
- ❌ `Phase 5 진행` — type/scope/내용 전부 부재

본문은 **비자명한 결정**에만 작성한다. "왜 이 토큰 이름을 골랐는가", "왜 이 파일만 제외했는가" 같은 선택 근거가 있을 때만. 자명한 변경이면 제목만으로 충분하다.

### 6-4. 수정 커밋: 스택 방식

사용자 확인 이후 수정 요청이 들어오면 **새 커밋을 쌓아서** 대응한다. `git commit --amend` / `rebase -i squash` / `reset` 으로 히스토리를 재작성하지 않는다.

- **이유**: 수정 커밋은 "판단이 교정된 사실"을 기록한다. 미래 세션의 AI가 log를 시간순으로 읽으면 "원 의도 → 교정 사유 → 최종 결과" 가 자연스럽게 복원된다. squash/amend는 이 맥락을 파괴한다.
- **메시지 규약**: 수정 커밋임이 제목 또는 본문에서 명시되어야 한다.
  - 제목 예: `fix(theme): Sidebar 토큰명 교정 (sizes.sidebar → sizes.sidebarWidth)`
  - 본문 예: `앞선 a1b2c3d 커밋에서 토큰명 오타. 정식 이름으로 교정.`

### 6-5. 체크리스트 작성 시 적용

Section 1의 `작업 내용` 체크리스트를 설계할 때, **각 항목이 커밋 1개와 자연스럽게 매핑되도록** 작성한다. 체크리스트가 곧 커밋 로그의 예상 모양이 된다.

- ❌ `- [ ] Phase 5: 고정 layout 치수 전부 토큰화`
- ✅
  - `- [ ] Sidebar 고정 layout 치수를 sizes 토큰으로 치환`
  - `- [ ] TaskBar 고정 layout 치수를 sizes 토큰으로 치환`
  - `- [ ] StartMenu 고정 layout 치수를 sizes 토큰으로 치환`

이 원칙을 따르면 플랜과 git log 가 1:1 로 정렬되어, 진행 상태 추적(`- [x]`)과 실제 커밋 로그가 같은 정보를 보여주게 된다.
