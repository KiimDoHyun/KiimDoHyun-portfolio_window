# 설계 문서 작성 가이드

설계 문서를 작성할 때 다음 패턴을 따른다.

## 1. Phase별 독립 실행 가능하게 설계

각 Phase는 **자기 완결적**으로 작성한다. 이전 대화 맥락 없이 문서만 읽고 바로 실행 가능해야 한다.

Phase마다 다음을 명시한다:

- **입력**: 이 Phase를 시작하기 위해 필요한 선행 조건
- **작업 내용**: 구체적인 변경 항목 (체크리스트) — 각 항목은 커밋 1개에 대응하도록 설계한다. 매핑 원칙은 [6. 체크리스트와 커밋의 매핑](#6-체크리스트와-커밋의-매핑), 커밋 규약 본체는 [`commit-convention.md`](commit-convention.md) 참조
- **완료 조건**: 이 Phase가 끝났다고 판단하는 기준

**완료 조건은 고정한다.** 문서에 명시한 뒤에는 작업 도중 임의로 수정하지 않는다. 수정이 필요하면 먼저 사용자에게 사유를 설명해 승인을 받은 뒤 문서에 반영하고, 그 후에 이어서 작업한다. 작업 진행 중 "어쩌다 보니 바뀌어 있는" 완료 조건은 플랜과 실제 작업의 계약을 무너뜨린다.

## 2. 배경(Why)과 규칙을 문서에 내장

매 세션마다 반복 지시하지 않아도 되도록, 문서 자체에 포함시킨다:

- **배경**: 왜 이 작업이 필요한지
- **설계 규칙**: 작업 중 지켜야 할 원칙 (예: "Shell에는 로직을 넣지 않는다")
- **성공 기준 (Definition of Done)**: 전체 작업의 완료 조건

## 3. 진행 상태 추적

`- [ ]` / `- [x]` 체크리스트를 사용하여 진행 상태를 문서에 직접 반영한다. 새 세션에서 "이어서 해줘"가 가능해야 한다.

## 4. Phase 완료 시 검증과 회고

각 Phase가 완료되면 다음 두 가지를 문서에 남긴다.

### 4-1. 수동 검증 항목

사용자가 직접 확인해야 할 테스트 시나리오를 구체적으로 작성한다. 나중에 자동화 테스트의 기반 데이터로 활용된다.

```markdown
#### 수동 검증
- [ ] 바탕화면 아이콘 더블클릭 → 해당 프로그램 창이 열린다
- [ ] 폴더 열기 → 하위 파일 목록이 정상 표시된다
- [ ] 상태바 → 프로젝트 목록이 올바르게 노출된다
```

### 4-2. Phase 회고

주관적 평가가 아니라 **검증 가능한 사실 기반 관찰**만 남긴다. 사용자가 검토 후 확정한다.

```markdown
<!-- 나쁜 예 (검증 불가) -->
> 패턴이 잘 적용되었다

<!-- 좋은 예 (검증 가능) -->
> fsState를 통째로 넘기다가 selector로 변경 → Shell 코드가 12줄 → 5줄로 줄었다
```

### 4-3. 검증 실패 시 처리

Phase 의 완료 조건을 만족하지 못하면 해당 단계를 반복한다. **같은 단계가 3회 이상 실패**하면 계속 시도하지 말고 사용자에게 상황을 보고한 뒤 방향을 협의한다. 반복 실패는 대개 완료 조건 자체가 현실과 어긋났거나, 설계 단계에서 보이지 않았던 제약이 드러난 신호다.

보고에는 다음을 포함한다:

- 시도한 접근들과 각각의 실패 원인
- 완료 조건을 재정의해야 할 것 같다면 그 제안
- 설계 문서(§1) 의 완료 조건을 그대로 둘지, 사용자 승인 하에 수정할지

## 5. 프로젝트 회고

모든 Phase 완료 후, Phase별 회고를 취합하여 전체 회고를 작성한다:

- **잘된 점**: 다음에도 유지할 패턴
- **개선할 점**: 다음에 보완할 사항
- **향후 과제**: 이 작업에서 파생된 후속 작업

반복 적용 가능한 교훈은 본 가이드(plan-writing-guide.md)에 반영하여 다음 프로젝트 설계에 자동으로 적용되도록 한다.

## 6. 체크리스트와 커밋의 매핑

커밋 단위 기준, 메시지 포맷, 수정 스택 방식 등 커밋 규약 전반은 [`commit-convention.md`](commit-convention.md) 를 따른다. 이 절에서는 **설계 문서 작성 관점에서만** 주의할 점을 다룬다.

Section 1 의 `작업 내용` 체크리스트를 설계할 때, **각 항목이 커밋 1개와 자연스럽게 매핑되도록** 작성한다. 체크리스트가 곧 커밋 로그의 예상 모양이 된다.

- ❌ `- [ ] Phase 5: 고정 layout 치수 전부 토큰화`
- ✅
  - `- [ ] Sidebar 고정 layout 치수를 sizes 토큰으로 치환`
  - `- [ ] TaskBar 고정 layout 치수를 sizes 토큰으로 치환`
  - `- [ ] StartMenu 고정 layout 치수를 sizes 토큰으로 치환`

이 원칙을 따르면 플랜과 git log 가 1:1 로 정렬되어, 진행 상태 추적(`- [x]`)과 실제 커밋 로그가 같은 정보를 보여주게 된다. 플랜을 작성할 때 체크리스트 한 줄이 [`commit-convention.md` §3 메시지 포맷](commit-convention.md#3-커밋-메시지-포맷) 에 그대로 끼워넣어도 말이 되는 모양이라면 매핑이 잘 된 것이다.
