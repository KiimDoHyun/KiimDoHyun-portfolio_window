# 협업 컨벤션

## 1. "추천" 항목은 엄격하게 검증

옵션을 제안할 때 `(추천)` 표기는 신중하게 사용한다. 사용자는 특별한 이변이 없는 한 추천 옵션을 그대로 채택한다.

**추천 전 자체 검증 체크리스트:**

- [ ] **장단점을 모두 나열했는가** — 추천 옵션의 단점도 명확히 인지했는가
- [ ] **대안을 진지하게 검토했는가** — 다른 옵션을 단순 비교용으로 깎아내리지 않았는가
- [ ] **현재 코드베이스의 제약**과 충돌하지 않는가 (예: 동적 마운트 구조, 기존 컨벤션)
- [ ] **이전 단계의 결정**과 일관적인가
- [ ] **다음 단계의 작업**을 어렵게 만들지 않는가
- [ ] **사용자가 명시적으로 언급한 선호**(과거 피드백, docs/rules)와 충돌하지 않는가
- [ ] **YAGNI/단순성** 관점에서 과도하지 않은가
- [ ] **되돌리기 비용**이 합리적인가

위 항목을 모두 충족하지 못하면 `(추천)`을 붙이지 않거나, 사용자에게 결정을 넘긴다.

**Why:** 사용자가 추천을 신뢰하고 그대로 따르기 때문에, 추천이 부실하면 큰 손실로 이어진다. "추천"은 책임 있는 단언이지 단순 의견이 아니다.

**How to apply:**
- 옵션을 제시할 때마다 추천 후보를 자체 검증
- 검증을 통과한 항목만 `(추천)` 표기
- 추천하는 이유에 단순 장점만 나열하지 말고, "왜 다른 옵션보다 나은가"를 명시
- 확신이 없으면 추천 없이 사용자에게 묻기
